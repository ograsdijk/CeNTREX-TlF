## Context

Two coupled tasks:

1. **Physics correction (Phase 0)**: the current simulator treats the RF as an **electric** field that couples via the molecular dipole (HS_α matrices). This is wrong for the CeNTREX Ramsey geometry — the RF is **magnetic, in x**, driving the Tl nuclear spin via g_Tl·μ_N·**B**·**I**. The hyperfine-induced electric-RF coupling we accidentally calibrated at 72.5 V/cm has the right transition frequency (119.64 kHz) and direction (Tl spin flip) by coincidence, but the wrong physical amplitude scale and the wrong polarization sensitivity. Order-of-magnitude target: g_Tl·μ_N ≈ 25 Hz/G ⇒ π/2 in 1.6 ms ⇒ B_π/2 ≈ 20 G. We need to wire B-field RF through the simulator (centrex_tlf's `H_func(E, B)` already accepts B in Gauss; the change is in our FieldStack / propagator plumbing) and redo the calibration.

2. **Performance investigation (Phases A–D)**: once the physics is correct, the simulator works at Jmax=3 but is too slow at Jmax=6 for parameter scans:

| Setting | per trajectory | 41-pt ω_rf scan |
|---|---|---|
| Jmax=3, segmented grid, dt_fine=0.5 µs | ~62 s | ~42 min |
| Jmax=6, segmented grid, dt_fine=0.5 µs | **~9 min** (estimated, eigh is 13× slower) | **~9 h** |

We need Jmax=6 for trustworthy results (J=2 mixing at 30 kV/cm is significant; the dressed J=1 manifold has ~3% J=3 admixture per the earlier `dressed_initial_states` diagnostics). The current bottleneck is `scipy.linalg.eigh(N=196)` at ~40 ms per call, called 13837 times per trajectory under the segmented grid.

This plan is to **investigate, prototype, and benchmark a set of speedup options**, then write a short report quantifying impact and feasibility of each. The deliverable is a recommended "fast path" that lets a Jmax=6 scan finish in <1 h on this machine (8+ cores, no GPU; CUDA available on a separate machine).

## Key facts from exploration (cite once, don't re-derive)

- **Sparsity at Jmax=6** ([`X_uncoupled.py:315-378`](../../centrex_tlf/hamiltonian/X_uncoupled.py#L315-L378)): full H is ~95% sparse (1790 / 38416 non-zeros). `Hff` block-diagonal in J. Stark `HSα` only couples ΔJ=±1.
- **`H_func` structure** ([`generate_hamiltonian.py:322-339`](../../centrex_tlf/hamiltonian/generate_hamiltonian.py#L322-L339)): just `2π·(Hff + Σ E_α·HS_α + Σ B_α·HZ_α)` — sub-matrices are pre-built; per-step assembly is ~1 ms (vs eigh 40 ms).
- **`centrex_tlf/effective_hamiltonian/`** ([`preparation.py`](../../centrex_tlf/effective_hamiltonian/preparation.py), [`solve.py`](../../centrex_tlf/effective_hamiltonian/solve.py)): is for OBE on optical transitions; the *technique* (basis alignment across field points, projection to a manifold) maps onto our problem but the implementation does not directly transfer. We re-implement a simpler X-state-only version in V3.
- **Existing batch infrastructure** ([`lindblad/batch.py`](../../centrex_tlf/lindblad/batch.py)): ODE-based, not applicable. We start from scratch for parallelism.
- **No JIT / GPU / Krylov anywhere in centrex_tlf**: clean slate.

## Phase 0 — Magnetic-RF correction (prerequisite to perf work)

The current code adds the RF E-field into `E_total` and calls `H_func(E_total, [0,0,0])`. To switch to magnetic RF in x:

1. **`fields.py`** — introduce a `MagneticRFRegion` dataclass that mirrors `AnalyticRFRegion` but with units of Gauss and an `envelope_vec(R)` returning a B-vector. Add `B_dc: Optional[AnalyticDCField] = None` and a `B_total(R, t)` method to `FieldStack` (defaults to zero, B_total = B_dc + Σᵢ cos(ωt+φ_i)·B_envᵢ). `AnalyticRFRegion` (electric) stays for now — eventually deprecated, but keep it to avoid breaking the validation script during Phase 0.

2. **`simulator.py`** — in `_make_H_at_t`: replace `H_func(E, zero_B)` with `H_func(fields.E_total(R, t), fields.B_total(R, t))`. Initial-state and detection diagonalizations also must use a B at the start position (zero unless B_dc set).

3. **`propagator.py:build_segmented_t_grid`** — extend the activity-detection loop to consider B-field gradients (`B_dc` derivatives) and B-RF envelopes too. Generalise the current "DC + RF" thresholds into "any-field-changing" detection.

4. **`demo_ramsey_rf.ipynb`** — update cells 5, 4c, 4d:
   - Replace each `AnalyticRFRegion.rounded_rectangle(amplitude=72.5 V/cm, …, direction=(1,0,0))` with `MagneticRFRegion.rounded_rectangle(amplitude≈20 G, …, direction=(1,0,0))`.
   - Re-run the single-pulse Rabi scan in Gauss (likely 5–50 G range; the resonance frequency 119.64 kHz should be unchanged because it's set by the molecular Hamiltonian, not the drive).
   - Confirm: the calibrated π/2 amplitude gives full 0→1 fringe contrast in the φ₂ scan, and the resonance scan shows fringe spacing 73.6 Hz.

5. **`validate_ramsey_rf.py`** — swap RF type, re-confirm all 6 numerical-invariant checks pass (fringe-contrast threshold may need adjustment).

**Effort**: 1 day including re-calibration. **Output**: a working magnetic-RF Ramsey simulation with the same demo geometry but physically correct drive. *This is the new Jmax=3 reference for the perf work in Phases A–D.*

## Variants to investigate (priority order)

### V1 — Baseline (current, post-Phase-0)
- Reference for all comparisons. Single trajectory at the demo geometry, Jmax=6, segmented grid, dt_fine=0.5 µs, RF on at calibrated π/2 (72.5 V/cm) at resonance (119.64 kHz), φ₂=π/2.
- Measured outputs: wall-clock per trajectory, per-step eigh time, survival, |Psi_final⟩.

### V2 — Process-level scan parallelism *(MUST-HAVE)*
- **What**: wrap `scan.run_scan` loop in `concurrent.futures.ProcessPoolExecutor`. Each scan point is independent and runs on its own process; basis QN and H_func sub-matrices serialize cheaply (one-shot pickle per worker startup).
- **File**: `ramsey_rf/scan.py` — add `n_workers: Optional[int] = None` (None → 1, "auto" → cpu_count − 1).
- **Expected**: ~Ncores× linear speedup (8 → 8×). Zero accuracy impact.
- **Effort**: ~50 lines + small test.
- **Test**: 41-point ω_rf scan, n_workers ∈ {1, 4, 8}; survival arrays must match bit-for-bit.

### V3 — Reduced-basis (truncated) propagator *(HIGH potential)*
- **What**: at the entrance field, pick the K eigenstates of `H_low` that span the "physically reachable" subspace from Psi0 — i.e. the J=1 dressed manifold (16 states) plus the few neighboring dressed sublevels strongly Stark-mixed with it (J=0/2/3 admixtures). Build a fixed K×N truncation matrix `T = V_low[:, selected]`. Propagate the projected state vector `c(t) = T† Psi(t)` of size K via `c(t+dt) = exp(-i (T† H(t) T) dt) c(t)`, then reconstruct `Psi_final = T c_final`.
- Two sub-flavors to compare:
  - **V3a** *static projection*: T fixed at entrance field (relies on adiabatic coupling staying inside the chosen manifold).
  - **V3b** *adiabatic-tracked projection*: build T at each grid step from the K eigenstates that adiabatically descend from the K chosen at entrance (essentially the same `adiabatic_dressed_initial_states` algorithm but applied to K states). More accurate near the strong-mixing crossover, more expensive (extra eigh).
- **File**: `ramsey_rf/propagator_truncated.py` — exposes a drop-in `propagate_midpoint_truncated(...)` with the same signature plus `subspace_size: int`.
- **Expected**: eigh cost drops as (K/N)³ — at K=32, N=196 → 230× per eigh, with overhead dropping it to ~50–100× realized speedup. Accuracy depends on K; needs a sweep K ∈ {16, 24, 32, 48} to find convergence.
- **Risk**: at the 30 kV/cm plateau the dressed J=1 manifold mixes 56% J=1, 14% J=0, 27% J=2, 3% J=3 — so K must include at least the corresponding dressed J=0/2/3 sublevels (~16+8+24+8 = 56 candidates; in practice the *dominantly* mixed ones, ~30, are enough).
- **Effort**: 1 day.
- **Test**: single trajectory at K = 16/24/32/48, compare survival and per-J vs V1 reference.

### V4 — Sparse + Krylov `expm_multiply`
- **What**: use `scipy.sparse.linalg.expm_multiply(-1j * H_sparse * dt, Psi)` instead of eigh + V·exp·V†. H is 95% sparse so the matvec is cheap; Krylov needs ~20–50 iterations per step.
- **File**: `ramsey_rf/propagator_krylov.py` — uses `scipy.sparse.csr_matrix` representation built from the centrex_tlf dense matrices.
- **Expected**: 2–5× per step. **Caveat**: midpoint with eigh is exact for constant H; Krylov adds truncation error (controlled by `tol`). For inactive single-step segments (12 ms, GHz-scale eigenvalues × 12 ms = 10⁹ rad), Krylov is likely worse than the current single eigh + diagonal exp.
- **Strategy**: use Krylov *only in active segments* of the segmented grid; keep eigh+exact for inactive segments. This needs a propagator that switches schemes based on the step type.
- **Effort**: 1 day, plus accuracy validation.
- **Test**: per-step time vs N (N=196 case), accuracy vs V1, sweep `tol` ∈ {1e-6, 1e-9, 1e-12}.

### V5 — Numba JIT for the per-step eigh-and-rotate hot loop
- **What**: factor the inner loop (V.conj().T @ Psi → multiply by exp(-iD dt) → V @ ...) into a Numba `@njit` function. eigh itself can't be JITed (calls LAPACK), but the per-step Python overhead disappears.
- **File**: `ramsey_rf/propagator_numba.py`.
- **Expected**: 5–15% (eigh dominates at N=196).
- **Effort**: half day. Cheap insurance.
- **Risk**: marginal at this N; more useful at N≤64.

### V6 — Lower-precision arithmetic (complex64)
- **What**: cast H, V, Psi to complex64 (single precision) before eigh. BLAS uses S/C drivers for float32; typically ~1.5–2× speedup for eigh, less memory bandwidth pressure.
- **File**: `ramsey_rf/propagator_lp.py`.
- **Expected**: 1.5–2× per step. Floating-point error per operation rises from 1e-15 to 1e-7; over 13800 steps could accumulate to ~1e-3 in survival. Quantify it.
- **Effort**: half day.
- **Test**: same as V1, report relative error on survival and Psi_final.

### V7 — GPU eigh via cupy *(test on the other machine)*
- **What**: `cupy.linalg.eigh` for individual matrices, OR batched `cupy.linalg.eigh` over `(n_scan, N, N)` for several scan points at once.
- **File**: `ramsey_rf/propagator_cupy.py` with optional import (try/except so the file is harmless without CUDA).
- **Expected**:
  - Single (N=196): ~2–3× over scipy after host↔device transfer overhead.
  - Batched 50 trajectories at once: ~10× per matrix, ~50× total throughput.
- **Effort**: 1 day to implement, then ship the bench script to the GPU machine and run.
- **Constraint**: cannot benchmark on this PC. Plan to test by running the same `perf_bench.py` script on the GPU machine and importing its results into the report.

### Lower-priority (note in report, don't test unless V2+V3 don't suffice):
- JAX (overlaps with cupy, larger refactor)
- Higher-order Magnus propagator (allows larger dt_fine — but the dt-convergence sweep already showed dt=0.5 µs is fine in segmented mode)
- Cross-scan-point intermediate-state caching (for φ₂ scans, the first half of each trajectory is identical) — useful if V2 isn't enough
- Tighter segmented-grid thresholds (currently 32% active; could trim to ~25%) — marginal

## Benchmark harness (`benchmarks/perf_bench.py`)

Single script, single configuration (Jmax=6 demo geometry, 72.5 V/cm, 119.64 kHz, φ₂=π/2):

```python
def run_one_variant(variant_name, propagate_fn, *, scan: bool = False) -> VariantResult:
    """Run the reference trajectory (and optionally a 41-point ω_rf scan) and
    return wall-clock + accuracy metrics vs the reference Psi_final."""
```

Variants register via a small dispatch table. Each emits:
- `time_per_traj_s`
- `time_per_step_ms`
- `survival`
- `max_abs_err_psi_final` vs reference (Jmax=6, dt_fine=0.25 µs, V1 baseline)
- `time_full_scan_min`
- `n_workers` / hardware notes
- `dependency_added`

The harness writes `PERF_REPORT.md` with one table summarising all variants and a short prose conclusion.

## Reference / "truth"

For each accuracy comparison: reference trajectory = **V1 baseline at Jmax=6, dt_fine=0.25 µs** (segmented grid). One-shot, ~18 min, cached to disk via `benchmarks/reference.py` (numpy `.npz` of `Psi_final`, t_grid, survival). All variants compare against this cached file.
